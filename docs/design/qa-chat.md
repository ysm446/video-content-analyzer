# チャット（動画 Q&A）の仕組みとレビュー

作成日: 2026-07-03
対象コード: `backend/server.py`（/review/qa）/ `backend/video_reviewer.py` / `frontend/pages/app.html`（チャットパネル）

## 1. 仕組み（2026-07-03 改善後）

```
質問送信（フロント）
  POST /review/qa {video_path, question, max_frames, min_interval, frame_mode,
                   transcript, history: 直近3ターンの [{question, answer}]}
  ↓
① VL モデルのロード（未ロード時のみ。分析後は常駐しているので通常はスキップ）
② フレームの取得（優先順）
   1. メモリキャッシュ（同一動画・同一パラメータで抽出済みならヒット）
   2. 分析キャッシュのサムネール（{動画名}.cache/data.json の scenes ＋
      thumbnails/*.jpg を復元。ffmpeg 不要で即開始。シーン4枚未満なら不採用）
   3. ffmpeg 抽出（uniform は各時刻への入力シーク×並列4。全編デコードしない）
   → 2/3 の結果はメモリキャッシュに保存し、次の質問から即応答
③ プロンプト構築（video_reviewer._build_qa_prompt）
   - フレーム画像（1枚 ≤256 視覚トークン、ctx 予算で自動間引き）＋ [m:ss] ラベル
   - [これまでのやり取り]: history の直近4ターン（回答は400字で切り詰め）
   - [字幕テキスト]: transcript から質問キーワード（2-gram）で関連行を3000字選別。
     一致がなければ全編から等間隔サンプリング
   - 質問文
④ llama-server にストリーミング推論 → answer_delta を SSE 中継
   （finish_reason / usage / tokens_per_sec を meta で返す。打ち切りは qa_warning）
⑤ フロントは Markdown 描画（sanitizeHtml 済み）し、完了時に state.qaChat へ
   {question, answer} を保存（最大6件、動画切り替え・履歴クリアでリセット）
```

## 2. 遅かった原因（今回の修正点）

**「チャット開始前の動画解析」の正体はフレーム抽出。**

- 従来、フレームのキャッシュはメモリ上の1スロットだけで、アプリ再起動後や
  分析とパラメータが揃わない場合は質問のたびに ffmpeg 抽出が走っていた
- しかも uniform 抽出は fps フィルタで**動画全編をデコード**する実装だったため、
  長い動画では抽出だけで数十秒かかっていた

対策（実装済み）:

1. **分析キャッシュのサムネール再利用** — 分析済みの動画は data.json の
   シーン（start_sec）とサムネール画像をそのまま QA フレームに使う。
   ffmpeg 起動ゼロで即回答生成に入れる。シーンサムネールは場面転換ごとの
   代表フレームなので、均等サンプリングより内容の網羅性もむしろ良い
2. **並列シーク抽出** — キャッシュが無い場合の uniform 抽出を
   「-ss 入力シーク × 1枚 × 並列4」に変更。全編デコードが不要になり、
   60秒動画で 8枚 0.13秒（実測）。長編ほど差が大きい

## 4. 既知の制約・今後の改善候補

- **時刻・チャプター依存の質問への密サンプリング**（video-analysis-review.md の 3-5）:
  「15:30 あたりで何をしてる？」等の質問時に `extract_frames_between()` で
  該当区間を密に取り直すと精度が上がる。未対応（任意）
- **プロンプト先頭（画像列）の KV キャッシュ**: cache_prompt=true を渡しているため、
  同じフレーム列での連続質問は画像部分のプレフィルが再利用されうる。
  ただし llama-server 側のマルチモーダルなプレフィックスキャッシュの効き方は
  バージョン依存のため、遅い場合はここを疑う
- **履歴の渡し方**: 現在は user メッセージ内のテキストブロック
  （翻訳と同じ「参考」形式）。会話が長くなるとトークンを圧迫するため
  直近3ターン・回答400字で制限している。要約履歴への切り替えは将来枠
- **シーンサムネールの解像度**: フロントのキャプチャ品質に依存する。
  低すぎる場合は `/cache/thumbnail` 保存時の品質を上げる余地がある
- **frame_mode=scene でキャッシュ不在の場合**は従来どおり scene 検出
  （全編スキャン）で遅い。分析キャッシュがあれば ② で回避される
